Overview

The ActiveRaUL system consists of two main components:
  • The RDFa User Interface Language (RaUL), a user interface markup ontology that can be used to describe the structure of a Web form as RDF statements.
  • The ActiveRaUL RESTful Web service, that coupled with a generic client servlet, generates a user interface model for semantic Web applications.

RaUL Ontology

The RDFa User Interface Language provides a standard set of visual controls that are similar to the form controls found in HTML. The RaUL markup separates the control elements of a web form from the data the form controls operate on. A data binding mechanism maps this data, expressed as RDF triples, to a form and its controls. The seperation of the form controls from the data makes RaUL forms more tractable by referencing values in the form controls with concepts defined in an ontology. It also eases reuse of forms, since the underlying essential part of a form (the data expressed in RDF) is no longer irretrievably bound to the page it is used in. Further, the RDF data maintained by the Web form can be accessed independently of the form.

Some of the main concepts in the RaUL widget ontology are:

  • raul:Page
  • raul:WidgetContainer
  • raul:Textbox
  • raul:Listbox
  • raul:Button

The full ontology can be found at: http://purl.org/NET/raul#.

ActiveRaUL

The ActiveRaUL service supports CRUD operations, such as storing and querying RDF data, on the underlying RDF data via HTTP requests. Additionally, the service provides a mechanism to produce and render web forms in RDF. These form instances can be rendered in many different RDF serialisations, among them, an XHTML+RDFa version that is used for display in the browser. A Client-side JavaScript RDFa API based on jQuery enables the direct interaction between the user (or more precisely the generated web forms) and the ActiveRaUL service. The API supports RDFa parsing of the annotated XHTML forms and encapsulates the communication with the backend service.

Using ActiveRaUL is easy, because:

  • ActiveRaUL loves resource-oriented (REST) APIs and their hypermedia links (see for the REST endpoints below).
  • You don't need new tools, since your browser can interact with ActiveRaUL.
  • Limited RDF knowledge required, because RaUL web forms can be generated from ontologies automatically. Also, standard RDF Web forms should become ubiquitous on the Linked data Web and ready for you to be reused.

Web Service Interface

The ActiveRaUL service offers HTTP resources to create, update and delete RaUL-based Web form widgets and their associated RDF data. The Web service manages the assignment of URIs to the submitted resources and returns the URL in the HTTP Location header of the response, for example, /public/forms/registeruser. We refer to these identifiers as formid in the table below. For the data submitted for a Web form, the ActiveRaUL service also assigns a URI, such as /public/forms/registeruser/arminhaller. We refer to these identifers as dataid in the table below.

The following table summarises all HTTP resources of the ActiveRaUL Web service together with their functionality and return values.

Create a formPOST/{userid}/forms

Read a data instanceGET/{userid}/forms/{formid}/{dataid}
